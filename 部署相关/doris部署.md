## 一、角色与核心组件

Doris 集群包含三类核心角色：

表格





| 角色              | 职责                           | 最低数量                   |
| :---------------- | :----------------------------- | :------------------------- |
| **FE (Frontend)** | 元数据管理、查询规划、集群调度 | 1（生产建议 3 节点高可用） |
| **BE (Backend)**  | 数据存储、查询执行             | 1（生产建议 3 节点起）     |
| **Broker**        | 外部数据源访问（HDFS、S3 等）  | 按需部署                   |

## 二、硬件配置建议

### 1. 测试/开发环境（单节点或最小集群）

表格





| 节点 | CPU  | 内存  | 磁盘       | 网络 |
| :--- | :--- | :---- | :--------- | :--- |
| FE   | 4 核 | 8 GB  | 100 GB SSD | 千兆 |
| BE   | 4 核 | 16 GB | 500 GB SSD | 千兆 |

> 单节点可同时部署 FE 和 BE，仅用于功能验证。

------

### 2. 生产环境（推荐 3FE + 3BE 起）

#### FE 节点配置

表格





| 规模   | CPU   | 内存     | 磁盘       | 说明             |
| :----- | :---- | :------- | :--------- | :--------------- |
| 中小型 | 8 核  | 16-32 GB | 200 GB SSD | 元数据内存密集型 |
| 大型   | 16 核 | 64 GB+   | 500 GB SSD | 高并发查询规划   |

**关键要求：**

- **内存优先**：FE 元数据全内存存储，需预留足够内存
- **SSD 必选**：元数据日志写入对磁盘 IOPS 敏感
- 生产环境至少 **3 个 FE 节点**（1 Leader + 2 Follower/Observer）

#### BE 节点配置

表格





| 规模   | CPU    | 内存       | 磁盘            | 说明              |
| :----- | :----- | :--------- | :-------------- | :---------------- |
| 中小型 | 16 核  | 64 GB      | 2-4 TB SSD      | 每节点            |
| 大型   | 32 核+ | 128-256 GB | 10 TB+ NVMe SSD | 或 HDD + SSD 缓存 |

**关键要求：**

- **内存**：查询中间结果、缓存、Compaction 均耗内存
- **磁盘**：强烈建议 **SSD**（HDD 仅适用于冷数据归档场景）
- **多磁盘**：BE 支持多路径存储，建议单节点挂载多块磁盘提升并行度
- **网络**：万兆网卡（高并发场景必备）

## 五、你当前情况的建议

结合你之前提到的 Doris 4.0.5 部署经验，如果你是要：

- **测试/学习**：1 台 8C/32GB 虚拟机即可同时跑 FE+BE

- **生产上线**：建议至少 **3 台物理机/云主机**，每台 16C/64GB/SSD，分别部署 FE 和 BE

- **接入现有业务**：根据你的 WAT/CP/FT 数据量级（百万级 WAT、36 万级 CP、6 万级 FT），3 节点 BE（16C/64GB/2TB SSD）已能很好支撑

  

 















Doris 的节点动态扩展**相对容易**，这是其核心设计优势之一。不同角色的扩展方式和难度有所区别：

------

## 一、BE 节点扩展（最容易）

BE 负责数据存储和计算，扩展最为平滑：

### 操作步骤

sql



```sql
-- 1. 在新节点启动 BE 服务
-- 2. 在 FE 中注册新 BE
ALTER SYSTEM ADD BACKEND "new_be_host:9050";

-- 3. 数据自动均衡（Doris 会自动触发）
-- 默认通过 Tablet 迁移实现负载均衡
```

### 关键特性

表格





| 特性         | 说明                                         |
| :----------- | :------------------------------------------- |
| **在线扩展** | 无需停服，业务无感知                         |
| **自动均衡** | 新 BE 加入后，FE 自动调度 Tablet 迁移        |
| **并行扩容** | 可同时添加多个 BE，加速均衡                  |
| **缩容支持** | `ALTER SYSTEM DECOMMISSION BACKEND` 安全下线 |

### 注意事项

- **均衡时间**：取决于数据量，TB 级数据可能需要数小时
- **资源影响**：Tablet 迁移会占用网络和磁盘 IO，建议低峰期操作
- **副本安全**：确保集群有足够副本（默认 3 副本），迁移过程中不影响查询

------

## 二、FE 节点扩展（中等难度）

FE 负责元数据，扩展需谨慎：

### 两种扩展方式

表格





| 类型         | 命令                                    | 用途                   | 投票权 |
| :----------- | :-------------------------------------- | :--------------------- | :----- |
| **Follower** | `ALTER SYSTEM ADD FOLLOWER "host:9010"` | 高可用，参与选举       | 有     |
| **Observer** | `ALTER SYSTEM ADD OBSERVER "host:9010"` | 扩展读能力，不参与选举 | 无     |

### 操作步骤

bash



```bash
# 1. 从现有 FE 复制元数据目录到新节点
# 2. 新节点启动时指定 --helper 指向现有 Leader
./start_fe.sh --helper leader_host:9010

# 3. 在 Leader FE 执行 SQL 添加
ALTER SYSTEM ADD FOLLOWER "new_fe_host:9010";
```

### 注意事项

- **元数据同步**：新 FE 需从 Leader 同步完整元数据镜像，启动时间较长
- **选举机制**：Follower 数量为 **奇数**（1/3/5），避免脑裂
- **Observer 更安全**：仅扩展查询能力，不影响选举，适合读密集型场景

------

## 三、Broker 扩展（最简单）

Broker 是无状态服务，直接启动即可：

sql



```sql
ALTER SYSTEM ADD BROKER broker_name "broker_host:8000";
```

------

## 四、扩容后的数据均衡策略

Doris 提供多种均衡控制手段：

sql



```sql
-- 查看均衡状态
SHOW PROC '/backends';

-- 调整均衡速度（避免影响业务）
ADMIN SET FRONTEND CONFIG ("tablet_rebalancer_type" = "BE");

-- 手动触发均衡（通常自动即可）
ADMIN REBALANCE TABLET ON BACKEND "be_host:9050";
```

## 五、你的场景建议

结合你之前提到的数据量级（WAT 百万级、CP 36 万级、FT 6 万级）：

表格





| 阶段       | 配置          | 扩展时机                 |
| :--------- | :------------ | :----------------------- |
| **起步**   | 1 FE + 3 BE   | 当前已足够               |
| **增长期** | 3 FE + 5 BE   | 查询变慢或磁盘 >70% 时   |
| **成熟期** | 3 FE + 10+ BE | 数据量破亿或并发查询增加 |

### 实际操作建议

1. **BE 按需扩容**：监控磁盘使用率和查询延迟，超过阈值即加节点
2. **FE 提前规划**：建议初期就部署 3 FE（1 Leader + 2 Follower），避免后期元数据迁移风险
3. **利用 Observer**：如果 BI 查询并发高，可单独加 Observer FE 分担查询压力
